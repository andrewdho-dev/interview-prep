01

45분 기술 인터뷰의 의미

채용담당자의 안내 메시지에서 세 가지 시그널을 읽어내야 한다.

  • 시간 · 45분 — 일반적 1차(60분)보다 짧다. 라포 빌딩 여유가 적고 질문 밀도가 높다는 뜻.
  • 성격 · 기술 인터뷰 — 메시지에서 명시. 일반 핏 검증보다 기술적 깊이 검증의 비중이 더 클 가능성이 높다.
  • 준비 가이드 · "JD 및 제출 서류 기반" — 자기소개서·이력서에 적은 모든 문장이 질문의 출처가 될 수 있다. 특히 구체 명사로 적은 것(Cognizant, ITC Infotech, T-shirt sizing, EAV, 9억, MAU 150만 등)은 그대로 검증 대상.

시간 분배 추정

  • 5~7분 · 자기소개 + 지원 동기
  • 25~30분 · 기술적 깊이 검증 (BAT 사례 deep-dive, JD-역량 매칭)
  • 8~12분 · 지원자 역질문
1시간 버전과의 차이 · 60분이었다면 "이력서를 똑같이 말로 반복하지 말라"는 일반론으로 충분했지만, 45분에선 모든 답변이 처음부터 trade-off / 의사결정 / 학습으로 들어가야 한다. STAR 형식의 길고 평이한 답변은 시간만 잡아먹고 임팩트가 약하다.

회사 제공 리소스 활용 전략

채용담당자가 4개의 리소스를 직접 보내준 것은 단순 "참고하세요"가 아니라 "이건 우리가 어떤 회사인지 보여주는 자료다, 인터뷰에서 이 맥락을 공유하고 시작하자"는 신호다. 면접 답변에 자연스럽게 인용할 수 있어야 한다.

예시 패턴:

"보내주신 Pleos 25 키노트에서 송창현 사장님이 [키워드]를 강조하시는 부분이 인상적이었는데, 이게 제가 BAT에서..."

이 한 문장이 깔리면 면접관은 즉시 "이 사람은 우리 자료를 봤고, 우리 언어로 자기 경험을 매핑할 줄 안다"는 판단을 하게 된다. 1시간 인터뷰에서는 굳이 안 해도 되는 동작이지만, 45분에서는 강력한 차별화 포인트.

2.4 42dot의 자기 정의

회사 공식 메타데이터·블로그에서 일관되게 쓰는 자기 정의는 "Mobility AI Company"다. "자율주행 회사"가 아니라 "AI 회사"로 자기 포지셔닝을 한다. 자체 LLM(42dot LLM 1.3B), Active Learning 파이프라인, Transformer 기반 인지 모델 등 AI 기술이 자율주행 / SDV / 음성 인터페이스 / 보안 등 모든 영역의 기반을 이룬다고 본다.

면접에서 회사를 지칭할 때 "자율주행 회사"보다 "모빌리티 AI 회사" 또는 그냥 "42dot이 만들고 있는 플랫폼"으로 부르는 게 더 정확하다. 작은 디테일이지만 회사를 이해하는 깊이의 시그널.

2.5 나의 narrative와의 연결고리

"OEM별/차종별 반복 인증 → 플랫폼 인증 표준화"는 내가 BAT에서 "시장별 반복 구축 → 글로벌 플랫폼 표준화"로 풀어온 문제와 정확히 같은 형태.
→ 이걸 면접 첫 5분 안에 자연스럽게 깔아야 한다.
03

Pleos Playground 그룹의 실체

채용담당자 안내 메시지에서 그룹 정의를 정확히 인용하면 다음과 같다.

"Pleos Playground 그룹은 Pleos App Market, Playground 데이터 제품, 그리고 이를 기반으로 한 다양한 서비스·비즈니스 운영을 포괄합니다."

즉 합류하면 다루게 될 영역은 세 갈래다.

① Pleos App Market

  • 차량용 앱을 개발자들이 직접 올리고, 차량에서 사용자가 다운로드해 쓰는 차량 전용 앱 스토어
  • 2025년 Pleos 25 컨퍼런스에서 사전 공개에 26개 개발사가 베타 앱 시연
  • 구글, 삼성(SmartThings), 네이버, 우버, 쏘카, 유니티가 글로벌 파트너로 참여
  • 모바일 앱스토어와 다른 점: 주행 안전 / 시선 분산 최소화 / NPU 자원 제약 / OTA 업데이트 등 차량 도메인 특수 제약이 인증 표준에 들어감

② Playground 데이터 제품

  • SDK·API·디자인 가이드·디버깅 도구·실차 없는 앱 테스트 환경 등 개발자 플랫폼 그 자체
  • Pleos 디자인 가이드는 Figma 파일로 공개되어 있으며 (pleos.ai/playground) 디자인 토큰·컴포넌트 시스템이 명세화되어 있음
  • JD의 "데이터 제품"은 곧 개발자 경험(DX) 제품이라고 이해해야 함

③ 서비스·비즈니스 운영

  • App Market과 Playground 위에서 돌아가는 실제 운영 비즈니스
  • 개발자 사용량·앱 매출·차량별 활성도 등 metric을 다루는 데이터 제품 영역과 자연스럽게 연결
핵심 통찰 · 이 그룹의 일은 완성차 제조가 아니라 플랫폼 비즈니스이다. Andrew의 BAT 경험(D2C 이커머스 + Adobe Commerce + 다중 벤더 + 글로벌 파트너)이 자동차 제조사 직접 경험보다 오히려 더 직결되는 영역이다. 이걸 인터뷰에서 분명히 인지하고 있어야 한다 — "저는 자동차를 모르지만"으로 시작하면 안 되고, "플랫폼 비즈니스 운영 경험을 가져왔습니다"로 시작해야 한다.

04

플랫폼 고도화 전략 · TPM 관점

왜 이 섹션이 필요한가 · Pleos Playground의 Develop 페이지를 직접 분석한 결과를 바탕으로, TPM 포지션에서 "이 플랫폼을 어떻게 고도화할 것인가"에 대한 구체적인 의견을 정리해둔 것이다. 면접관이 "합류하면 어떤 일을 하고 싶으세요?" 또는 "Playground를 살펴봤다면 어떤 개선이 필요하다고 보셨어요?"라고 물었을 때 바로 꺼낼 수 있는 무기.

Pleos Playground는 현대자동차그룹이 만든 차량용 앱 개발자 플랫폼이다. 스마트폰 앱을 만들려면 Android/iOS 개발 환경이 필요하듯, 차량 화면에서 돌아가는 앱을 만들려면 차량 전용 개발 도구가 필요한데 그걸 모아둔 곳이 Playground다.

현재 제공하는 핵심 도구는 네 가지다.

도구역할
Pleos Connect SDK차량 기능(주행 데이터, 내비, 음성인식, ADAS 등)을 앱에서 사용할 수 있게 해주는 코드 묶음. Vehicle SDK, NaviHelper SDK, Gleo AI SDK, ADAS SDK로 구성.
Fleet API렌터카·물류회사 등이 여러 대의 차량 상태를 실시간 모니터링할 수 있는 API.
Vehicle Data API현대·기아·제네시스 차량 데이터를 표준화해서 하나의 인터페이스로 조회.
Devbox & Emulator실제 차량 없이도 앱을 테스트할 수 있는 개발 환경.
TPM의 본질적 역할 · 이 도구들이 더 잘 굴러가게 만들고, 개발자가 만든 앱이 실제 차량에 들어가기까지의 검증 과정을 빠르고 안전하게 다듬는 일이 바로 Platform TPM의 핵심 과제다.

1단계 · 플랫폼 아키텍처 고도화

짚어두기 · Pleos Playground는 이미 Vehicle SDK / NaviHelper SDK / Fused Location SDK로 분리해 제공하고 있다 — 이건 이미 modular SDK 설계가 아키텍처 차원에서 구현된 상태라는 뜻. "SDK를 쪼개자"는 제안은 회사가 이미 하고 있는 일을 모르고 하는 말이 된다. 외부에서 보고 "구조를 파악했다"고 말하려면 한 단계 더 들어가야 한다.

진짜 어려운 문제 — SDK 간 버전 호환성 매트릭스

SDK를 쪼갠 이후에 실제로 어려워지는 건 "임의로 조합된 SDK 버전이 임의의 차종·OS 버전에서 제대로 동작하는지 보장"하는 일이다. Vehicle SDK v1.2 + NaviHelper SDK v0.8 + Fused Location SDK v0.1 조합을 26개 베타 개발사가 각자 다른 버전으로 쓰기 시작하면, 인증해야 할 차종 × SDK 버전 × OS 버전의 cross-product가 폭발적으로 늘어난다.

  • 차종별로 인증을 반복하면 2030년 2,000만 대 목표를 맞출 수 없다 — JD의 "Platform-level Certification"이 풀려는 진짜 문제가 이 차원일 가능성이 높다.
  • 이 문제를 다루려면 "SDK를 쪼개는 설계" 자체가 아니라 공통 코어 레이어의 추상화 수준 설정, 버전 호환성 매트릭스 관리, SDK 간 의존성 그래프가 핵심 과제다.

SDK upgrade migration 경로 — 차량의 OTA 주기를 고려해야 하는 이유

모바일 생태계에서도 SDK v1 → v2 migration이 가장 어렵다고 꼽히는 이유는, 이미 출시된 앱들의 하위 호환성을 유지하면서 구버전을 퇴장시켜야 하기 때문이다. 차량은 이게 더 까다롭다 — OTA 업데이트 주기가 모바일보다 길고, 차종별 OTA 정책도 다르다. 어떤 차종은 OTA가 제조사 서비스 센터 방문 시에만 이루어지는 경우도 있다.

BAT 경험 매핑 · BAT에서 Adobe Commerce 컴포넌트 40종의 deprecated 경로를 정의하고 "v1 → v2에서 무엇이 깨지는지" Cognizant 파트너 팀과 협상한 것이 같은 구조. 다만 차량은 OTA 주기 제약이 더 강하다는 차이가 있고, 그 차이가 migration 전략 전체를 바꾼다.

글로벌 표준과의 연결고리 — API 표준 통합

Fleet API와 Vehicle Data API가 현대·기아·제네시스 데이터를 표준화한 건 강점이지만, 글로벌 OEM 확장을 고려하면 국제 차량 데이터 표준(ISO 20078 / ExVe, CVII, COVESA VSS)과의 매핑 레이어가 필요하다. 외부 모빌리티 서비스 사업자가 "Pleos에 한 번만 연결하면 여러 브랜드 차량에 다 통한다"고 느낄 수 있어야 한다.

클라우드 테스트 환경 — Cloud Devbox

지금의 Devbox·Emulator 외에 클라우드 기반 가상 차량 테스트 환경을 추가하면 —

  • 물리 하드웨어 없이 브라우저에서 바로 가상 차량 환경에 접속해 테스트. 진입 장벽이 확 낮아진다.
  • CI/CD 파이프라인(GitHub Actions, GitLab CI)과 연결되면 코드 수정 때마다 자동으로 차량 환경 테스트가 돌아가는 흐름을 만들 수 있다.
  • 공식 Action 배포 → 자동화된 빌드–테스트–시뮬레이션 워크플로우 완성.

2단계 · 인증 절차를 4단계 관문으로 정리

차량용 앱은 일반 앱과 다르다 — 잘못 만들면 운전자의 안전과 직결된다. 그래서 검증이 꼼꼼해야 하는데, 외부에서 어떤 항목을 어떻게 보는지 명확히 보이지 않을 가능성이 크다. 이걸 4단계 게이트 모델로 정리하자는 제안이다.

관문검증 대상방법
1관문 · 정적 검증SDK 사용 규약, API 호출 제한, 매니페스트 권한, 라이선스 컴플라이언스자동 스캐너 — 5분 내 완료 가능
2관문 · 기능 검증시나리오별 기능 동작(내비 목적지 입력 등), Vehicle HAL 호환성Emulator 기반 자동 테스트
3관문 · 안전·UX 검증주행 중 산만도(Driver Distraction), NHTSA·UNECE 권고, 음성·터치 응답성자동 측정 + 사람 리뷰 병행
4관문 · 보안 검증정적/동적 분석, 펜테스트, 개인정보 동의 흐름보안 전문 리뷰 + 자동화 도구

이렇게 단계를 나누면 개발자가 어디서 막혔는지 명확히 알 수 있고, 검증팀도 일을 분담하기 쉬워진다.

3단계 · 검증을 빠르게 만드는 방법

자동화를 점진적으로 확대

  • 첫 6개월 → 1관문(정적 검증) 100% 자동화
  • 다음 6개월 → 2관문(기능 테스트) 80% 자동화
  • 1년 내 → 3·4관문(안전·보안) 70% 자동화
  • 결과 → 평균 리뷰 소요 시간 현재 대비 50~70% 단축 목표

제출 전 자가 진단 도구 — Pre-Submission Self-Check

개발자가 앱을 제출하기 전에 "내 앱이 통과할 가능성이 몇 퍼센트인가" 미리 확인할 수 있는 도구. 시험 전 모의고사와 같다. 처음부터 통과할 만한 앱만 제출되니 반려율(Rejection Rate)이 떨어지고, 검증팀 부담도 줄어든다.

구조화된 피드백

검증에서 떨어졌을 때 "안 됨"이 아니라 "코드 어느 줄에서 어떤 규칙을 위반했고, 이렇게 고치면 됩니다"라고 구체적으로 알려준다. 재제출 사이클이 짧아진다.

BAT 경험 매핑 · 일본 myGlo UAT에서 결함을 'business logic 위반 / UX 일관성 / edge case'로 분류하고 각각 release blocker 여부를 정의한 것과 같은 구조.

신뢰 등급제 — Trusted Partner Track

  • Standard Track · 신규 개발사 → 4단계 전 검증을 모두 받는다.
  • Trusted Partner Track · 여러 번 통과한 검증된 파트너 → 일부 검증을 Self-Attestation("본인이 확인했다"는 서명)으로 대체.
  • 효과 · 한정된 검증 인력을 신규 개발사에 더 집중할 수 있다.

4단계 · 실행 거버넌스

정기 운영 체계

  • 분기별 Platform Roadmap Review — 로드맵 점검
  • SDK Council — SDK 오너 + DevRel + Security + Safety가 모이는 협의체
  • Reviewer Calibration Session — 검증 담당자끼리 "같은 앱을 보면 같은 판정이 나오는지" 보정

  • Early Access → 일부 파트너 대상 공개, 피드백 수집

5단계 · CI/CD 파이프라인 & 배포 전략

JD 직접 연결 · Gerrit·Jira 자동화 경험이 JD에 명시되어 있고, 차량 OTA는 웹 서비스와 구조적으로 다르다. "간단한 CI/CD 알아요"와 "차량 OTA의 특수성을 감안한 CD를 설계할 수 있어요"는 전혀 다른 답변이다.

Gerrit 기반 자동화 — 5개 도메인 특화 검증

단순 빌드 성공 여부를 넘어, 개발자가 Gerrit에 Push하면 파이프라인이 즉각 다음을 검증한다.

  • 정상 빌드 + 기본 단위 테스트
  • 이기종 시스템(AAOS, ECU) Cascading impact · Breaking change 여부
  • 차량 제어 기능 ms 단위 Latency 테스트
  • 보안 코딩 규정(Secure coding compliance) 충족 여부
  • Driver distraction 방지 기준 (주행 중 미디어 UI 비활성화 등)
5가지 통과 시 CI 서버가 Gerrit에 자동으로 Verified +1을 부여하고 Jira 티켓 상태를 동기화. 사람 리뷰 리소스는 아키텍처·비즈니스 로직에만 집중할 수 있는 구조.

OEM 양산 기준을 충족하는 4단계 CD 파이프라인

차량은 웹 서비스처럼 즉각 롤백이 어렵다. 리스크를 가장 앞단에서 발견하는 것이 목표.

단계환경검증 목적
① SIL클라우드 가상 머신순수 소프트웨어 로직 검증
② HIL연구소 랙 (Head Unit + ECU)차종별 하드웨어 제약 타겟 테스트
③ Test Fleet주행 시험장 실차실제 차량 엣지 케이스 점검
④ Canary사내 → 1% → 10% → 100%점진적 OTA Roll-out · 리스크 분산

Canary 단계 치명적 이슈 발생 시 — 두 층위 동시 가동

  • CD 레벨 (Circuit Breaker) · 모니터링에서 에러율 임계치 초과 감지 → Webhook으로 파이프라인에 신호 → 남은 99% 차량 OTA 자동 중단(Halt)
  • 차량 레벨 (Dual-Bank OTA) · 하드웨어/OS 레벨의 A/B 파티션 — 시스템 멈춤 시 안전했던 파티션으로 자동 Fallback. 네트워크 없이도 복구 가능.
면접 답변 핵심 · "즉각 롤백"이 어렵다는 제약 조건을 먼저 인정하고, 그래서 앞단 게이트가 왜 중요한지와 차량 레벨 Fallback 아키텍처가 왜 필수인지를 두 층위로 설명하는 것이 완전한 답변.

Shift-Left 전략 — 버그를 파이프라인 앞단으로

치명적 버그 RCA 후 — 해당 차종 하드웨어 세팅을 HIL 장비에 추가 연동 → 엣지 케이스를 테스트 스크립트화 → 전체 Regression Test 슈트에 편입. "버그 수정"이 아닌 "파이프라인 강화"가 TPM의 언어.

한 줄 요약

인증은 4단계 관문으로 명확히 하고, 자동화와 자가 진단 도구로 속도를 높이며, 신뢰 등급제로 효율을 챙긴다. 그 바탕에서 진짜 platform-level 문제 — SDK 버전 호환성 매트릭스, OTA를 감안한 migration 경로, 26개 개발사 DX 마찰점의 분류 — 를 정의하는 것이 TPM의 고유 역할.

면접에서 이 지식을 꺼내는 방법

옵션 A — 통째로 빼기 (안전) · "합류하면 이런 걸 하겠다" 류의 답변은 45분 1차에서 위험. 회사가 이미 하고 있는 일을 "개선안"으로 제시하면 역효과. 경험·trade-off 사고로만 답하고, 이 가설은 면접관이 직접 물을 때만 꺼낼 것.
옵션 B — Discovery 질문으로 전환 (더 강한 시그널) · "SDK가 이미 modular하게 분리되어 있다면, platform-level 표준화가 풀려는 진짜 어려운 문제는 SDK 간 버전 호환성 매트릭스나 OTA를 감안한 migration 경로 같은 차원일 것 같습니다. 실제로 내부에서 가장 critical하게 보고 있는 platform-level 문제가 어느 쪽인가요?" → "내가 답을 안다"가 아닌 "내가 문제 구조를 정확히 인지한다"는 시그널. 이게 실제 TPM의 일.
04-B

개발자 생태계 개선 아이디어

이 섹션의 성격 · Pleos 25 키노트를 직접 보고 떠오른 나의 관점. 면접에서 "플랫폼을 어떻게 고도화할 것인가" 질문이 나왔을 때 꺼낼 수 있는 차별화 인사이트다. 섹션 04의 TPM 기술 전략과 함께 읽을 것.

① 개발자 온보딩 — 차량 플랫폼 특유의 투명성 문제

Pleos Playground는 신생 마켓플레이스다. 삼성의 타이젠·바다OS처럼 초반에 대형 파트너를 유치했더라도, 결국 생태계를 만드는 건 개인 개발자와 중소 개발사의 트래픽이다. 그런데 차량은 모바일과 근본적으로 다른 제약이 있다 — 하드웨어 교체 주기가 극히 길다.

개발자 입장에서 앱 기획 단계에 가장 먼저 알고 싶은 것은 다음이다.

개발자의 질문현재 Playground에서 찾을 수 있는가?
내 앱이 지원되는 차종·모델·연식명확하지 않음
해당 차종의 예상 등록 대수 (시장 크기)공개 안 됨
OS 메이저 업그레이드 지원 횟수 (SLA)정책 없음
SDK 버전 지원 종료(EOL) 일정명확하지 않음

"이 차량군을 타겟으로 앱을 만들면 향후 4년간 50만 명 유저에게 닿을 수 있고, OS 메이저 업그레이드 2회까지 내 앱이 지원된다" 는 계산이 가능해야 처음부터 투자 결정이 선다. 이 정보가 없으면 기획 단계에서 이미 포기하게 된다.

TPM 액션 · 차종별 예상 등록 대수 로드맵 + SDK/OS 버전 지원 SLA를 개발자 센터의 상시 공개 정보로 만드는 것. 카카오디벨로퍼스식 구성에서 차량 특유의 하드웨어 수명주기 투명성으로 차별화.

② 결제 생태계 — 앱스토어가 성공한 진짜 이유

대형 개발사 입장에서 앱을 만드는 이유는 자사 DB 확보와 결제 수익, 개인 개발자 입장에서는 결제 수익과 광고다. 앱스토어 생태계가 성공한 핵심 이유 중 하나는 플랫폼 초창기부터 비교적 심리스한 결제 연동을 투명한 수익 셰어 모델과 함께 제공했기 때문이다.

  • Pleos에는 인카 페이먼트(In-Car Payment) 인프라가 있다 — 애플페이에 대항하는 포지셔닝이 가능한 자산.
  • 레거시 문제 · ccNC 등 구버전 OS에 얽힌 결제 레거시를 Pleos 생태계로 어떻게 마이그레이션할지가 핵심 과제.
  • 개발자 입장에서 명확한 수익 셰어 정책(예: 앱 내 결제의 X%, 구독 Y%)이 공개되지 않으면 앱 개발 ROI를 계산할 수 없다.
TPM 액션 방향 · 인카 페이먼트 API를 Playground SDK에 공식 통합 + 결제 수익 셰어 정책을 Day 1부터 투명하게 공개. 결제가 붙어야 개발자 생태계가 돌아간다.

③ 에뮬레이터 — Xcode 전략에서 배울 것

"소프트웨어 테스트는 에뮬레이터, 실차 연동은 DevBox"라는 방향은 맞다. 그런데 개발 생태계 표준화의 핵심은 에뮬레이터가 어떤 차량군(Group)을 시뮬레이션하는가다.

Xcode가 사용성이 나쁘다는 평가를 받음에도 개발자들이 포기 못 하는 이유는 단 하나 — "어느 세대, 어떤 디바이스와 호환되고 어디서 끊기는지" 정확히 확인 가능하기 때문이다.

"이 기능은 iOS 16 이상에서만" 자동 경고"이 SDK API는 Pleos 2.x 이상 차량군에서만" 자동 경고
단서 · 레거시 차량(SDV 이전)은 호환성 보장이 구조적으로 어렵다. 애플이 구버전 iOS 지원을 과감히 끊듯, Pleos 탑재 차량부터만 이 호환성 그룹화 체계를 적용하는 것이 현실적이고 올바른 선택일 수 있다.
면접 활용법 · 이 세 가지 아이디어를 "제가 합류하면 이걸 하겠다"로 꺼내면 위험. 대신 역질문 또는 "Pleos 25를 보고 생각한 것"을 물을 때 꺼낼 것. "개발자 입장에서 온보딩 정보 투명성이 어느 수준인지 궁금한데, 내부에서 이미 계획 중인 부분이 있나요?" 형태가 더 강하다.

역질문 5개

면접관이 hiring manager일 가능성이 높으니, "이 자리에서만 답할 수 있는 질문"을 던져야 한다. 검색하면 나오는 질문은 절대 금지.

① 조직 구조 이해

"Pleos Playground 그룹과 AVP 본부의 일은 어떻게 분장되나요? Platform TPM이 두 조직 사이에서 어느 쪽 의사결정에 더 가깝게 일하게 되나요?"

→ 회사를 공부하고 왔다는 시그널 + 본인이 어느 layer에서 일할지 사전 파악

② 양산 일정의 현실

"2026년 2분기 첫 신차 출시 일정이 이미 발표된 상황인데, 이 시점에 합류하는 TPM이 첫 분기에 가장 critical하게 처리해야 할 일이 무엇인지 가늠이 가능할까요?"

→ "내가 합류했을 때 뭘 해야 하나"를 직접 묻는 것. 실무 감각 시그널

③ 표준화의 실제 진척도

"JD에서 말씀하신 'Platform-level Certification 체계' 전환은 현재 어느 정도 단계에 있나요? 이미 몇 개 차종에 시도 중인지, 아니면 첫 정의 단계인지 궁금합니다."

→ 회사가 이 일을 0에서 시작하는지, 일부 진행 중인지에 따라 합류 후 일이 완전히 다르다

④ 글로벌 파트너 협업의 현실

"구글 AAOS, 네이버, 삼성 SmartThings 같은 외부 파트너와의 기술 협상이 일상적인 업무라고 들었는데, TPM이 이런 협상에 단독으로 들어가나요, 아니면 PM/엔지니어와 팀으로 들어가나요?"

⑤ 평가 / 성장 (마지막 즈음)

"이 포지션의 첫 6개월 성과는 어떻게 평가되나요? 회사가 이 자리에서 어떤 시그널을 보고 'leveling up할 수 있는 사람이다'라고 판단하시는지 궁금합니다."
절대 묻지 말 것 · 연봉 / 복지 / 휴가 (1차에서 자살) · 회사 홈페이지에 적힌 사실 확인 ("재택근무 되나요?") · 상장 계획 / 주가 전망 (정치적)

08

면접관이 평가하고 있을 3가지

면접관은 1시간 내내 머릿속으로 다음 세 가지를 채점하고 있다.

  1. 이 사람이 우리 문제를 이해하는가?
    → 회사 컨텍스트 + JD를 자기 언어로 재진술 가능?
  2. 이 사람이 우리 팀에서 같이 일하기에 무리가 없는가?
    → 자기 약점을 인지? trade-off 사고가 가능? 자기 결정에 self-criticism 가능?
  3. 이 사람을 채용하는 risk가 무엇인가?
    → 연차 미달 / 도메인 미경험 — 이걸 어떻게 mitigate하는지 본인이 답을 가지고 있는가?
답변 하나하나가 이 세 질문 중 하나를 깎거나 쌓고 있다는 감각으로 임할 것.

마지막 한 가지 — 톤

42dot은 (현대 모기업 대비) 스타트업 결의 회사다. 면접관이 어느 결을 띠는지는 첫 5분에 보일 텐데, 상대 톤에 맞춰 거울 효과로 따라가는 게 안전하다. 스타트업 톤이면 자기소개도 더 캐주얼하게, 대기업 톤이면 더 formal하게.

다만 나의 강점은 "솔직하고 구체적"이라는 점. 어느 톤이든 그 강점은 유지되어야 한다. 톤은 표면, 콘텐츠는 본질.

카드를 탭하면 모범 답변이 펼쳐진다. 먼저 질문만 보고 머릿속에서 답을 만든 뒤, 카드를 펼쳐 비교하는 순서로 학습할 것. 학습 완료를 누르면 진행도가 저장된다 (브라우저 로컬에만).
면접 직전 30분에 한 번 훑을 카드들. 키워드 칩을 탭하면 짧은 설명이 뜬다.
